# Example output — abd-context-app-extractor

Each page folder contains three files produced automatically by `extract.spec.ts`.

```
pages/
  01-example-page/
    aria.yaml            ← semantic structure (roles, headings, labels, links, buttons)
    screenshot.png       ← full-page visual (not included here — binary)
    interactivity.json   ← raw DOM interactivity facts (cursor:pointer + ARIA states)
```

---

## aria.yaml

Playwright's `page.ariaSnapshot()` — the accessibility tree expressed as YAML.

**What it captures well:**
- Semantic roles: `heading`, `button`, `link`, `navigation`, `main`, `banner`
- Text content of interactive elements
- Structural nesting where the HTML uses semantic landmarks

**What it misses:**
- Non-semantic clickable elements (divs/spans with `cursor:pointer`)
- Active/selected state expressed via CSS classes (e.g. `plan-card--active`)
- Carousel dots, icon-only arrows, and other visually obvious affordances
  that lack ARIA roles
- Which tab/slide/item is currently selected when selection is CSS-only

See `aria.yaml` in this folder for an annotated example.

---

## interactivity.json

Generated by `captureInteractivity()` via `page.evaluate()` — raw DOM facts.

**Two sections:**

### `clickable_non_semantic`

Divs, spans, and list items that have `cursor:pointer` but are NOT natively
interactive elements (`button`, `a`, `input`, etc.).

These are invisible to ARIA but fully clickable. Common patterns:
- Card components that wrap a button
- Carousel dots / pagination indicators
- Custom icon-only nav items
- Row click handlers in tables

Each entry:
```json
{
  "tag": "DIV",           ← element tag
  "text": "Basic ...",   ← truncated text content (≤80 chars)
  "classes": "plan-card", ← space-joined class list (deduplicated by class fingerprint)
  "role": null,           ← explicit role= attribute, if any
  "testid": "plan-basic", ← data-testid, if present
  "count": 1              ← how many elements share this exact class fingerprint
}
```

When `count > 1`, the same component renders multiple times (e.g. 3 plan cards,
5 nav items). The `text` is from the first instance.

### `aria_states`

Elements that carry a non-false ARIA state attribute at capture time:
`aria-selected`, `aria-current`, `aria-expanded`, `aria-checked`,
`aria-pressed`, `aria-disabled`.

Use this to know **which item is active/selected** at page load — the ARIA
snapshot shows the tree structure but not which node is currently active.

Each entry:
```json
{
  "attribute": "aria-current", ← which attribute triggered this
  "value": "page",             ← its current value
  "tag": "A",
  "text": "Services",
  "role": "link"
}
```

**What interactivity.json does NOT do:**
- Interpret meaning — that's downstream (story mapping, UX spec)
- List all elements — only non-semantic clickables and active ARIA states
- Capture hover states or animation — only the DOM snapshot at load time

## Reading them together

| Question | Source |
|---|---|
| What is the page structure? | `aria.yaml` |
| What are the headings and labels? | `aria.yaml` |
| What buttons and links exist? | `aria.yaml` |
| Are there hidden clickable zones? | `interactivity.json → clickable_non_semantic` |
| Which tab / item is currently selected? | `interactivity.json → aria_states` |
| Is a menu expanded or collapsed? | `interactivity.json → aria_states` |
| What does the page look like visually? | `screenshot.png` |
