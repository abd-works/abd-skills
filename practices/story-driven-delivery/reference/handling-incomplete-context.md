# Handling Incomplete Context (shared)

Cross-cutting discipline for story-driven-delivery skills (story mapping, acceptance criteria, specification by example, and the rest). Skills link here instead of repeating it.

## Assess context coverage before producing anything

Before mapping, writing acceptance criteria, or drafting scenarios, check whether the available context covers the relevant dimensions:

- Business / product intent.
- Technical constraints.
- Operational concerns.
- Regulatory or compliance requirements (where applicable).
- User experience.
- Testing concerns.

If an entire dimension is absent, flag it as a gap rather than quietly proceeding as if it were covered.

## Don't fabricate to fill gaps

When context is incomplete, do not invent structure, stories, criteria, or scenarios to make the output look finished. For each gap:

- Capture what is missing.
- State the assumption you are making.
- Recommend the validation action that would resolve it.

Unknowns surfaced honestly are more valuable than a plausible-looking artifact that turns out to be wrong.

## Multiple perspectives, not solo work

These artifacts are meant to be read by product, delivery, and domain people at once. When context from a key perspective is missing, flag it as an unknown rather than substituting a single viewpoint for the whole picture.
