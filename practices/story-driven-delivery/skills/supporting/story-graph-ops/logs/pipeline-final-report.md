**Import fix-ups made:** none required for a move (there was nothing to move). Every co-located spec file uses an absolute-import bootstrap:

## Behaviors not translated / architecture conflicts

- **Rename confidence score.** `bdd-context.md` says the UpdateReport records rename with a confidence score. `UpdateReport.add_rename` accepts a `confidence: float | None` argument and currently emits `1.0` for every position-based fallback match. The spec asserts only that it is non-`None`; a heuristic scoring algorithm can plug in later without an API change.
- **Empty Epics in language backends.** `## Code` in `bdd-context.md` implies every Epic re-renders. In the TypeScript and Java backends, an Epic with no SubEpics produces zero files (nothing to emit). This is a true property of the target code layer, not an under-tested one; sync tests for those backends therefore assert edits at the SubEpic layer. Python's per-Epic `<epic_snake>_helper.py` gives the Python backend an Epic-level file, so its parse recovers empty Epics correctly. The abstract `CodeStoryMap.parse` was tightened during Section 4 to also recognise Epic-level helper files at depth 3 — a genuine bug surfaced by these sync tests.
- **Draw.io / Miro placement compression.** Section 3 collapses several BDD leaves that share an underlying invariant into fewer `it` blocks (e.g. "the first four Epics should keep their previous X positions" — one block, not one per Epic). Every distinct claim is still exercised; only the wording is deduplicated. Detailed in `logs/pipeline-diagrams.md`.
- **Sync in the format layer.** Sections 2, 3, 4 all implement `sync()` by delegating to `canonical.translate_from(parsed)` — the pure-domain translate_from from Section 1. Under the layer-isolation rule ("mocks only at architectural boundaries"), the format layer depending on the domain layer without mocking is correct; the domain layer is where translate_from is exhaustively covered.
- **`abd-clean-code` SKILL.md.** The pipeline prompt lists an `abd-clean-code` skill, but `c:\dev\abd-skills\practices\architecture-centric-engineering\skills\abd-clean-code\SKILL.md` does not exist. Clean-code discipline was applied inline from the abd-bdd-development rules (`code-minimalism`, `layer-isolation`, `oo-api-design`, `observable-behavior`) plus standard practice: single-responsibility private helpers, guard clauses, constructor injection, domain vocabulary, `WHY`-only comments. Each section log's `abd-clean-code applied` (or per-rule) table records the verdict.
