# Test Helpers and Domain Test Objects

This folder holds the test infrastructure for pml-my: MSW handlers shared across all three tiers, the `with-providers` wrapper for component and hook tests, and the per-sub-epic folders that contain the actual scenarios, domain test objects, and spec files. Tests are generated by the `abd-story-acceptance-test` skill; this file documents the layout, principles, and conventions the generated tests follow.

### Principles

- **One scenario definition, three execution tiers.** Each sub-epic has a single `*-scenarios.ts` file naming the given/when/then triples; T1 (Presentation), T2 (Application), and T3 (E2E) adapters execute the same scenarios in different runtime modes.
- **Build vertically, one acceptance criterion at a time.** Write the T1 test for AC #1 -> make it pass -> commit; then T2 -> pass -> commit; then T3 -> pass -> commit; then move to AC #2. Never write all T1s first across every AC.
- **Stub only at the tier boundary.** T1 and T2 stub HTTP via MSW; T3 stubs midtier via Playwright MSW handlers or test-account fixtures. `vi.mock` on HTTP modules is an anti-pattern.
- **Domain test objects wrap existing hooks; they do not create new production code.** Each domain test object maps 1:1 to a domain class in `docs/domain/model/domain-specification.md` and internally calls the real production hook (via `renderHook`) which then hits MSW stubs.
- **Helpers own mechanics; test files own scenarios.** Test files contain only `it` / `test` declarations and assertions; the scenario file owns the data; the domain test object owns the wiring.

### Testing Scope

| Tier | Entry point                  | Real                                            | Stubbed                              | Asserts                                          | Proves                              |
| ---- | ---------------------------- | ----------------------------------------------- | ------------------------------------ | ------------------------------------------------ | ----------------------------------- |
| T1   | `render(<Component />)`      | React, hooks, Recoil, MUI                       | HTTP (MSW), Cognito, GrowthBook      | DOM state, atom values, navigation calls         | UI wiring                           |
| T2   | domain test object method    | hooks, Recoil, Axios, `ApiReturn`               | HTTP (MSW), Cognito                  | object state, API call sequence (MSW spy)        | event -> API -> state               |
| T3   | Playwright `page.goto`       | full SPA stack including Amplify (test pool)    | pml-midtier via MSW or fixtures      | page content, navigation, network                | everything works in a real browser  |

### Story to Test Naming

| Story artefact         | Test artefact                                      |
| ---------------------- | -------------------------------------------------- |
| Epic                   | top-level folder                                   |
| Sub-epic               | subfolder inside the epic folder                   |
| Story                  | `describe` block inside the spec file              |
| Acceptance criterion   | `it` / `test` method inside the `describe`         |

### File Naming Convention

`<sub-epic-kebab>-<role>.ext` -- role is one of `-scenarios.ts`, `-domain-objects.ts`, `-presentation.spec.tsx`, `-application.spec.ts`, `-e2e.spec.ts`.

### Domain Test Object Shape

Each domain test object lives at `tests/<epic>/<sub-epic>/<sub-epic>-domain-objects.ts`, maps 1:1 to a class in `docs/domain/model/domain-specification.md`, exposes only the operations under test for the sub-epic, internally renders the relevant production hook via `renderHook`, and returns inspectable state (atom values, MSW call history). Domain test objects are test infrastructure; they never become production code.
