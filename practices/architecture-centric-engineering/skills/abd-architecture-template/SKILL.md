# abd-architecture-template

## Purpose

Turn a stable `architecture-context.md` (mechanism-tier — File Structure, Participants, Class Specification, Rules, Canonical Patterns) into a **runnable parameterized reference module** on disk. The output is the thing `abd-architecture-code` resolves as `<spec-root>` and copies-and-renames per feature — preserving everything the legacy reference applications (catalog-hero, Recipients, hero-vtt, MERN-domain-first) used to give the code skill, but sourced from the new project-distributed specification.

---

## Output shape

A **template package** at `docs/architecture/templates/<slug>/` containing:

| Subdirectory | Content |
|---|---|
| `README.md` | What this package instantiates, parameter list, the rename map, runbook for the example. |
| `template/` | The runnable reference module — real source files using placeholder names (`{System}`, `{Feature}`, `{DomainName}`) verbatim from the spec's Canonical Patterns. |
| `templates/tests/` | Parameterized test files, one per tier defined by the test-helpers package-tier `architecture-context.md`. |
| `example/` | One concrete instantiation of the template (e.g. `PartnerA` for Partner Integrations) — compiles, tests pass. |
| `rules/` | Mechanism-specific code-shape rules lifted verbatim from the source `architecture-context.md` § Rules. |
| `parameters.json` | Declarative list of placeholders the template defines, with binding hints. |

This is the `<spec-root>/template/`, `<spec-root>/templates/tests/`, `<spec-root>/example/`, `<spec-root>/rules/` shape that `abd-architecture-code` already knows how to consume.

---

## Modes

| Mode | Output location | When |
|---|---|---|
| `project` *(default)* | `docs/architecture/templates/<project-slug>/` — one package covering the project's primary mechanism (the most-instantiated one). | Most projects. Matches the legacy reference-application shape (one canonical scaffold per project). |
| `mechanism` *(opt-in)* | `docs/architecture/templates/<mechanism-slug>/` — one package per mechanism, generated by re-running the skill once per named mechanism. | Projects with multiple distinct mechanisms whose recipes diverge enough that one shared scaffold loses fidelity (e.g. Partner Integrations and Caching live side-by-side and have nothing to share). |

Both modes produce the same internal package structure; only the location and the choice of which mechanism to scaffold change.
