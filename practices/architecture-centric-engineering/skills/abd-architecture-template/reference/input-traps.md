# Input traps — abd-architecture-template

Assumptions, ambiguities, and missing context that commonly produce a broken or misleading template package. Check each trap against available input before generating — flag gaps honestly; route back to `abd-architecture-specification` rather than inventing missing spec content.

- **Spec gaps in the source mechanism** — does the source `architecture-context.md` actually carry filled File Structure, Participants, Class Specification, Rules, and Canonical Patterns? `<!-- spec to fill -->` markers mean the spec is not ready and the template would have to invent design decisions. Stop and route back.
- **Placeholder vocabulary drift** — the source spec uses `{Domain}` in Canonical Patterns; the test-helpers context file uses `{Entity}`; the existing project example uses `{Model}`. Pick the spec's vocabulary verbatim and surface the drift back to the spec author. Do not silently normalise.
- **Mode-vs-mechanism mismatch** — `project` mode on a system with three unrelated mechanisms forces a choice (which mechanism gets the canonical template?) that is not in the spec. Either ask the user, or argue for `mechanism` mode instead. Do not pick silently.
- **Sentinel collision** — the chosen sentinel for `example/` is also the name of a real feature in the project (e.g. binding to `PartnerA` when `PartnerA` is a live integration). The example tests will conflict with real code. Pick a sentinel that is unmistakeably unreal (`ExampleCo`, `SampleCo`, `Sentinel`).
- **Test-helpers context file is the wrong fidelity** — the file exists but is at exploration-level fidelity (one paragraph rather than the full layer-to-tech table, folder structure, spec-alignment table, principles). The template's `templates/tests/` cannot be derived from a one-paragraph pointer. Route back to spec.
- **Toolchain assumptions** — the project mixes languages (`.ts` for frontend, `.cs` for backend, `.py` for scripts). Which one does `example/` live in? The source mechanism's spec usually answers this via File Structure file extensions; if it does not, ask.
- **Rule discovery scope** — the source `architecture-context.md` § Rules has five bullets; the practice-skill `abd-architecture-specification/rules/` has fifty. Only the former belong in `template/rules/`. The practice rules govern how to author the spec, not how to generate code from it. Do not lift practice-skill rules into the template package.
- **Across the Codebase deviations as authority** — the source spec's "Across the Codebase" table lists three instances with two deviations. Treat the canonical pattern (not the deviations) as the template's source. Deviations belong in the spec, not in the template.
- **`example/` runnability without project context** — `example/` builds against the project toolchain. Confirm before authoring that the project actually has a working build and test command — if not, the example will fail through no fault of the template.
- **Idempotency expectations on first run vs re-run** — first run scaffolds everything; re-run produces a drift report and updates only the spec-derived sections. Conflating the two leads to overwriting manual sentinel adjustments in `example/`. Always check whether the package already exists before writing.
