# Experiment Plan — Chain-of-Layer Pipeline

Branch: `solution_run_two`
Corpus: `test/mm3/solution/context/` (Mutants & Masterminds 3rd Edition)

---

## What This Experiment Is

A new pipeline approach replacing the old scanning-based extraction. Inspired by Chain-of-Layer (CIKM 2024) and AutoSchemaKG's EDC framework.

**Core principle:** Discover taxonomy layer by layer, top-down, with evidence indexed as you go. Each layer is a separate AI pass. Scanners enforce structure mechanically. AI resolves violations — scanners never propose fixes.

**Two parallel artifacts produced at every step:**
- **Domain model** — modules, concepts, properties, operations (what things are and own)
- **Interaction tree** — epics, sub-epics, stories (what actors do and what changes)

These are two views of the same coin and must be produced simultaneously.

### Step 2 — Deepen Concepts and Stories Per Module/Epic

**What it produces:**
- Full concept definitions: all properties typed, all operations with signatures, invariants where evidenced
- Full story definitions: Trigger, Response, Pre-Condition, Failure-Modes (up to 3)
- Hierarchy within modules: parent-child relationships between concepts derived from shared mechanics
- Sub-epics within epics: groupings of related stories
- Resolution of all `[defer]` flags from Step 1 via targeted chunk reads

**Inputs:**
- `step1-output-v3.json` — forward index from Step 1
- `step1-chunk-index-v3.json` — reverse index: tells Step 2 exactly which chunks to read for each concept
- Provisional chunk buckets from Step 1 — read these first for each module
- Individual chunk `.md` files in `context/`

**Approach:**
- One AI pass per module/epic pair (Chain-of-Layer — layer by layer, not all at once)
- For each module: read the identified and provisional chunks from the reverse index, then deepen concepts
- For each epic: deepen stories to full Trigger/Response/Pre-Condition
- Resolve `[defer]` flags: read the deferred chunks, decide subtype vs enum, name subtypes if warranted

**Output:**
- `step2-output.json` — deepened forward index (same shape as step1, more populated)
- `step2-readable.md` — human-readable summary
- Updated reverse index

**Quality passes:**
- All Step 1 scanners re-run on step2 output
- Additional scanners to design:
  - `scan_concepts_have_owns.py` — every concept must have `owns` (not just chunk_ids)
  - `scan_stories_have_trigger_response.py` — every story must have trigger and response
  - `scan_no_orphan_concepts.py` — every concept must appear in at least one story trigger or response

**Human gate:** Review module concept hierarchies and sub-epic groupings before Step 3.

### Step 3 — Canonicalize Across All Modules

**What it produces:**
- Unified, deduplicated model: same concept named consistently across all modules
- Cross-module relationships explicitly wired: e.g. PowerLevel constraining both Character and Powers
- Hierarchy finalized: subtypes named and placed, enums confirmed
- All `[cross-cutting]` flags resolved: each cross-cutting concept assigned to its home module with explicit references in other modules

**Inputs:**
- `step2-output.json`
- `step2-chunk-index.json`

**Approach:**
- One AI pass across the full output — not per module
- Look for naming inconsistencies, duplicate concepts across modules, orphaned relationships
- Wire cross-cutting concepts: Resolution (Check, Degree) referenced by every module
- Finalize variant families: confirm subtype hierarchies or enums from Step 2 decisions

**Output:**
- `step3-output.json` — canonical, clean forward index
- `step3-readable.md`
- Updated reverse index

**Human gate:** Approve canonical model before evidence extraction.

### Step 4 — Evidence Extraction (Code)

**What it produces:**
- Structured evidence per concept: actions (subject → verb → object), decisions (if/when/unless rules), states (lifecycle, transitions), relationships (explicit associations)
- Guided by Step 3 concept list — only extracts for named concepts, not noise

**Approach:**
- Existing code pipeline adapted from the old system (`evidence_extraction_guided.py` or equivalent)
- Run per concept: for each concept, read its `chunk_ids` from the Step 3 output and extract structured evidence from those chunks only
- No re-scanning of the full corpus — the chunk index tells the extractor exactly where to look

**Output:**
- `evidence/actions.json`, `decisions.json`, `states.json`, `relationships.json`
- Per-concept evidence records anchored to chunk_ids already in the index

**Quality:**
- Scanner: every action subject/object must reference a concept in the Step 3 output
- Scanner: no orphan evidence (evidence records that reference no concept)

### Step 5 — Structure (Model Generation)

**What it produces:**
- `solution_model.json` v1 — full domain model and interaction tree with evidence links
- Each concept: properties typed, operations with invariants, relationships with behavioral justification
- Each story: Trigger, Response, Pre-Condition, Failure-Modes, domain concept references
- Evidence refs linking concepts and stories to actions/decisions/states

**Inputs:**
- `step3-output.json` (canonical concept and story scaffold)
- `evidence/` (structured evidence from Step 4)

**Approach:**
- AI reads evidence per concept and deepens properties/operations from what evidence says
- AI links stories to behaviors (operations that implement them)
- AI assigns decision ownership: which concept owns each decision from `decisions.json`
- No invention — everything derived from evidence

**Quality passes:**
- Run existing `pipeline.py scan structure` (existing scanners)
- Run existing `pipeline.py validate structure` (adversarial pass)

### Step 6 — Behavior, Variation, Consolidate, Assess, Finalize

**Status: Unchanged from existing pipeline.**

These phases operate on `solution_model.json` and are not affected by the new Step 1–4 approach. The only difference is that the input model will be cleaner and better evidenced.

- **Behavior** — assign operations, link behaviors to steps, group steps into scenarios
- **Variation** — split stories by subtype when mechanics differ; add failure modes
- **Consolidate** — fix anti-patterns (anemia, over-centralization); add examples
- **Assess** — produce model assessment: consistency, coverage, completeness
- **Finalize** — apply assessment fixes; produce validated model

## Key Design Decisions Made

1. **Chain-of-Layer structure** — discover layer by layer, not flat extraction. Each step is a separate pass.
2. **Dual-view simultaneous** — modules and epics always produced together; never one without the other.
3. **Evidence-first** — every field that makes a claim cites a chunk. No citation = `[defer]` flag or blank.
4. **Scanners highlight, AI resolves** — scanners never propose fixes. AI determines: genuine gap | false positive | needs flag.
5. **Rule-scanner 1:1 map** — every scanner has a corresponding rule file with the same `rule_id`. Rule references scanner; scanner references rule.
6. **Junk config is solution-specific** — universal patterns in `junk_defaults.json` (skills level), domain-specific junk in `junk_config.json` (solution level). No domain terms in code.
7. **Stories are a sanity check at Step 1** — not the deliverable. 2–3 names to confirm the epic is real. Full story detail comes in Step 2.
8. **Flagging system** — `[defer]`, `[uncertain]`, `[cross-cutting]` are explicit outputs, not omissions.
