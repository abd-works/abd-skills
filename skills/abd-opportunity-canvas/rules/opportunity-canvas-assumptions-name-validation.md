# Rule: Canvas states testable assumptions with a validation hint

**Scanner:** Manual review

A filled opportunity canvas makes uncertainty legible: each **ASSUMPTION:** line names something that could be wrong and points to how you would learn fast. **Passing** means at least one **`ASSUMPTION:`** line includes a **`validate by:`** phrase (or equivalent “how we would know” wording on the same line). **Failing** means only solution ideas with no belief that could fail, or assumptions with no validation path when this practice already treats **making validation explicit** as the bar.

## DO

- Include at least one **`ASSUMPTION:`** line with a **`validate by:`** clause that a team could schedule (interview, prototype, data pull, pilot).

  **Example (pass):** `ASSUMPTION: Dealers expose an open slot API in Q2 — validate by: API spec review workshop with two dealer IT leads and a signed pilot scope.`

## DO NOT

- Leave **`ASSUMPTION:`** lines that are only opinions with no check.

  **Example (fail):** `ASSUMPTION: Users will love it` with no validation mechanism on the same or adjacent line in the canvas file.

**Source:** Agile by Design facilitator decks on the opportunity canvas (assumptions and validation explicit).
